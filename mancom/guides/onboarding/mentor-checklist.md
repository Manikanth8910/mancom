# Mentor Checklist

Checklist for mentoring new team members.

## Before Day 1

- [ ] Grant GitHub organization access
- [ ] Share Appwrite project credentials
- [ ] Add to Slack channels
- [ ] Assign a starter task (simple, well-defined)
- [ ] Schedule daily check-ins for first week

## Day 1 Check-In

Verify they've completed:

- [ ] Prerequisites installed
- [ ] Repository cloned
- [ ] `make setup` successful
- [ ] `make dev` runs
- [ ] Health endpoint works

Address any setup blockers immediately.

## Day 2 Check-In

Discuss:

- [ ] What they learned from code exploration
- [ ] Any confusing concepts
- [ ] Questions about architecture

Review their answers to the "Trace a Request" exercise.

## Day 3 Check-In

Support their first PR:

- [ ] Help them understand the task
- [ ] Review their branch naming
- [ ] Check their commit messages
- [ ] Provide constructive PR feedback
- [ ] Help them address feedback
- [ ] Approve and merge their first PR

Celebrate the first merge!

## Week 1 Topics to Cover

- [ ] How we use Git (branching, rebasing)
- [ ] How to run and write tests
- [ ] How to read error messages
- [ ] When to ask for help vs. research

## Ongoing Support

### 1:1 Meetings

Weekly for first month, then bi-weekly.

Cover:
- What they're working on
- What's blocking them
- What they've learned
- Career development

### Code Reviews

For first month:
- Review all their PRs promptly
- Explain the "why" behind feedback
- Point to relevant documentation
- Pair program on complex issues

### Pair Programming

Schedule pair sessions for:
- First implementation of a new pattern
- Debugging complex issues
- Understanding existing code

## Signs They're Ready for Independence

- [ ] Creates branches and PRs without help
- [ ] Writes tests without reminders
- [ ] Asks specific questions (not "it doesn't work")
- [ ] Finds answers in documentation
- [ ] Helps other new team members
- [ ] Suggests improvements

## Common New Developer Issues

| Issue | Solution |
|-------|----------|
| Afraid to ask questions | Explicitly encourage questions |
| Makes giant PRs | Set PR size expectations early |
| Doesn't write tests | Require tests in PR reviews |
| Copies code without understanding | Ask them to explain their code |
| Overcomplicated solutions | Show simpler approaches |

## Feedback Templates

### Positive Feedback

> "Great job on [specific thing]. I especially liked how you [detail]. Keep doing this."

### Constructive Feedback

> "This works, but let's improve it. Instead of [current approach], try [better approach] because [reason]. Here's an example: [link or code]."

### When They're Struggling

> "This is a tricky area. Let's pair on it. I struggled with the same thing when I started."

## Escalation

If an intern is struggling after 2 weeks:

1. Document specific issues
2. Discuss with their manager
3. Create an improvement plan
4. Provide extra support
5. Re-evaluate after another week
